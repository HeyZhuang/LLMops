## 问题1: langchain-core 版本漂移导致后端无法启动

**现象**：`python -m app.http.app` 报错 `ModuleNotFoundError: No module named 'langchain_core.pydantic_v1'`

**排查过程**：
- 报错定位到 `conversation_entity.py` 的 `from langchain_core.pydantic_v1 import BaseModel`
- 检查发现 `langchain-core` 实际版本 `1.2.18`，但 `requirements.txt` 锁定的是 `0.2.41`
- `langchain-core>=0.3` 移除了 `pydantic_v1` 兼容模块
- 追查到 `langchain-huggingface` 从 `0.0.3` 被升级到 `1.2.1`，间接拉升了 `langchain-core`

**修复**：
- 回退 `langchain-core==0.2.41`、`langchain-huggingface==0.0.3`、`langsmith==0.1.129`
- 新增 `constraints.txt` 约束文件，锁定 langchain 全系列版本上限
- 配置 `.venv/pip.ini` 自动加载约束，阻止未来任何 `pip install` 意外升级

**面试话术**：
> 项目中遇到过 Python 依赖版本漂移导致的生产事故。某次 pip install 间接升级了 langchain-core 大版本，破坏了 33 个文件的 pydantic_v1 导入。我通过 pip constraints 机制建立了版本约束体系，在 venv 级别自动生效，从根源上防止了依赖漂移。

## 问题2: Celery 启动命令与实际模块路径不一致

**现象**：`celery -A app.celery worker` 报错 `Unable to load celery application. The module app.celery was not found.`

**排查过程**：
- 查看代码发现 Celery 实例定义在 `app/http/app.py` 第 41 行：`celery = app.extensions["celery"]`
- 文档中记录的 `-A app.celery` 路径错误

**修复**：
- 正确命令：`celery -A app.http.app:celery worker --loglevel=info --pool=solo`
- 同步更新了 `CLAUDE.md` 文档

## 问题3: huggingface_hub 版本漂移导致 Embedding 调用失败

**现象**：知识库召回测试返回 `'InferenceClient' object has no attribute 'post'`

**排查过程**：
- `huggingface_hub` 从 `0.23.4` 被升级到 `0.36.2`
- `0.31+` 版本移除了 `InferenceClient.post()` 方法
- `langchain-huggingface==0.0.3` 的 `HuggingFaceEndpointEmbeddings` 内部依赖 `.post()`

**修复**：
- 回退 `huggingface-hub==0.23.4`
- 在 `constraints.txt` 中增加 `huggingface-hub>=0.23.0,<0.31.0` 约束

## 问题4: HuggingFace Inference API 端点迁移（核心难点）

**现象**：
- 旧端点 `api-inference.huggingface.co` → **410 Gone**（已停服）
- 新端点 `router.huggingface.co` → **404 Not Found**（URL 格式不兼容）

**排查过程**：
- 旧版 `huggingface_hub==0.23.4` 拼接的 URL：`/pipeline/feature-extraction/{model}` → 404
- 新版 `huggingface_hub==0.28.1` 的 `.post()` 内部逻辑完全重写，与 `langchain-huggingface==0.0.3` 传参方式不兼容
- 形成「旧库 + 新端点」和「新库 + 旧接口」的双重死锁

**修复方案 — 自定义 Embeddings 类**：
- 编写 `HuggingFaceRouterEmbeddings`，实现 LangChain `Embeddings` 接口
- 用 `requests` 直接 POST `https://router.huggingface.co/hf-inference/models/{model}`
- 完全绕开 `huggingface_hub` / `langchain-huggingface` 的版本依赖
- 通过测试发现正确的 API 路径需要 `/hf-inference/models/` 前缀

```python
class HuggingFaceRouterEmbeddings(Embeddings):
    """直接调用 HuggingFace Router API，不依赖 huggingface_hub 版本"""
    def _post(self, payload):
        url = f"{self.api_url}/hf-inference/models/{self.model}"
        response = requests.post(url, headers=self.headers, json=payload, timeout=30)
        response.raise_for_status()
        return response.json()
```

**面试话术**：
> 项目中遇到了 HuggingFace 停服旧 API 端点的情况，新旧端点与客户端库版本之间形成了兼容性死锁。我的解决思路是：既然三方库的版本约束无法同时满足，就跳出依赖，用 requests 直接调 HTTP API。自定义了一个实现 LangChain Embeddings 接口的类，通过逐一测试不同 URL 路径找到了新端点的正确格式（`/hf-inference/models/{model}`），彻底解耦了 embedding 调用与底层库版本。

## 问题5: Weaviate 向量数据库 Schema 缺少过滤属性

**现象**：`no such prop with name 'dataset_id' found in class 'Dataset' in the schema`

**排查过程**：
- 检查 Weaviate `Dataset` 集合，发现只有 `text` 一个属性
- 代码中 `SemanticRetriever` 按 `dataset_id`、`document_enabled`、`segment_enabled` 过滤
- 集合是空的（0 条数据），metadata 属性从未被自动创建

**修复**：
- 删除旧集合，重建时预定义全部 8 个属性：`text`、`dataset_id`、`document_id`、`segment_id`、`account_id`、`node_id`、`document_enabled`、`segment_enabled`
- 验证带过滤条件的向量查询不再报错

**面试话术**：
> Weaviate 虽然支持 auto-schema，但在集合为空时 metadata 属性尚未创建，查询过滤就会报错。我的做法是在初始化时预定义完整 schema，确保即使集合为空也能正常查询，这是向量数据库运维的一个最佳实践。

## 问题6: PDF 解析器在 Windows 环境下返回空内容

**现象**：文档状态 `completed`，但 `character_count=0`、`token_count=0`、0 个片段

**排查过程**：
- 从 COS 下载文件正常（270KB）
- `UnstructuredPDFLoader(strategy='fast')` 返回 0 字符
- `PyPDFLoader` 正常提取 1077 字符 / 2 页
- `Unstructured` 的 fast 策略在 Windows 上依赖链复杂，解析失败但没有抛异常

**修复**：
- `file_extractor.py` 中将 PDF 加载器从 `UnstructuredPDFLoader` 改为 `PyPDFLoader`
- 重新上传文档后成功解析：4816 字符 → 9 个片段 → 全部写入 Weaviate

**面试话术**：
> 文档处理流水线中发现 PDF 解析返回空内容但没有报错，属于静默失败。通过手动逐步测试（下载→解析→分块）定位到 UnstructuredPDFLoader 在 Windows 环境下的 fast 策略兼容性问题，换用 PyPDFLoader 解决。这个经验说明异步任务中的静默失败比显式报错更危险，需要对关键步骤做结果校验。
