# 医院影像 MVP 任务清单

## 目标版本

版本名称：`Imaging MVP v0.1`

聚焦场景：`胸部 CT 影像报告智能助手`

## 里程碑拆分

### M1：基础设施准备

- 梳理影像数据来源和脱敏规则
- 确认 PACS / RIS / 手工上传三类入口优先级
- 搭建 `Imaging` 业务域代码骨架
- 完成影像实体、状态枚举和工作流模板定义

验收：

- 仓库具备影像业务域基础结构
- 影像对象模型、状态流转和模板清晰可读

### M2：影像中心前端

- 新增影像中心规划页
- 展示目标场景、流程、MVP 边界和执行节奏
- 补充后续可扩展的数据契约类型

验收：

- 可通过前端路由访问影像中心页面
- 页面可直接用于方案演示与需求评审

### M3：后端数据层

- 新增检查、序列、实例、AI 结果、报告、反馈、审计模型
- 统一状态枚举和值域
- 预留推理结果和报告版本字段

验收：

- 数据模型能够覆盖从导入到审核的完整链路
- 模型字段支持后续数据库迁移设计

### M4：工作流模板

- 新增胸部 CT 报告助手模板
- 新增头颅 CT 高危分诊模板
- 模板输出统一结构化约定

验收：

- 工作流模板可直接作为产品和研发对齐依据

### M5：执行准备

- 形成接口清单
- 形成实施清单
- 形成第一阶段真实医院试点范围说明

验收：

- 产品、研发、算法、实施可以按清单并行推进

## 第一阶段接口清单

1. 影像接入接口
   - 上传 DICOM
   - 查询检查列表
   - 查询检查详情

2. 影像处理接口
   - 创建分析任务
   - 查询分析结果
   - 获取结构化发现

3. 报告接口
   - 生成报告草稿
   - 保存医生修改版
   - 确认正式报告

4. 审计接口
   - 查询访问日志
   - 查询审核日志
   - 查询反馈统计

## 团队执行建议

1. 后端先完成数据模型、状态设计和任务流转骨架
2. 前端同步完成影像中心、报告工作台和详情页原型
3. 算法以“独立推理服务”方式接入，不耦合主 Flask 进程
4. 产品和实施尽快完成一个医院试点的真实流程访谈

## 风险清单

- 影像数据样本获取慢，导致模型验证滞后
- PACS / RIS 对接周期不可控
- 医生对 AI 草稿接受度依赖模板质量
- 合规边界如果定义不清，后续试点容易卡住

## 下一步建议

1. 先用当前仓库的影像域骨架推进数据库迁移设计
2. 再补充后端 Handler / Service / Schema 的真实接口
3. 然后接入 OHIF 或 Cornerstone 完成影像浏览器
4. 最后接肺结节或头颅 CT 的首个专病模型
