## 1. 后端整体架构

**关键文件**：`app/http/app.py`、`internal/server/http.py`

**亮点**：自定义 `Http` 类继承 Flask，在构造函数中完成所有初始化——扩展注册、异常处理、中间件绑定、路由注册一步到位。

```
Http.__init__() 流程：
1. 加载 Config 配置对象
2. 注册全局异常处理器（CustomException返回结构化JSON，生产环境兜底500）
3. 初始化 Flask 扩展：SQLAlchemy → Migrate → Redis → Celery → Logging → LoginManager
4. CORS 跨域配置（允许前端5173/3000端口）
5. 注册 Middleware 的 request_loader（认证入口）
6. 注册 Router（路由蓝图）
```

**异常处理设计**：所有业务异常继承 `CustomException`，携带 code/message/data。全局处理器统一捕获，开发环境直接抛出便于调试，生产环境返回标准 JSON 响应。

## 2. 依赖注入体系

**关键文件**：`app/http/module.py`

**亮点**：使用 `injector` 库实现 IoC 容器，所有 Handler/Service/Middleware 通过 `@inject @dataclass` 自动装配依赖，无需手动传参。

```python
# ExtensionModule 绑定基础设施
class ExtensionModule(Module):
    def configure(self, binder):
        binder.bind(SQLAlchemy, to=db, scope=singleton)
        binder.bind(Redis, to=redis_client, scope=singleton)
        binder.bind(LoginManager, to=login_manager, scope=singleton)

# 全局注入器
injector = Injector([ExtensionModule])

# 使用示例——任何类只需声明类型，自动注入
@inject
@dataclass
class AppService:
    db: SQLAlchemy          # 自动注入
    redis: Redis            # 自动注入
    cos_service: CosService # 自动注入（级联依赖也自动解析）
```

**好处**：31个Service、21个Handler的依赖关系全部自动管理，新增服务只需声明类型字段。

## 3. 双蓝图认证机制

**关键文件**：`internal/router/router.py`、`internal/middleware/middleware.py`

**亮点**：一个 Router 注册两个 Flask 蓝图，Middleware 根据请求所属蓝图自动选择认证方式。

```
┌─────────────────────────────────────────────┐
│ 请求进入                                      │
│   ↓                                          │
│ Middleware.request_loader(request)            │
│   ├─ blueprint == "llmops"                   │
│   │   └─ JWT Bearer Token 认证 → Account     │
│   │      （前端用户登录后的Token）               │
│   └─ blueprint == "openapi"                  │
│       └─ API Key 认证 → Account              │
│          （第三方系统集成的密钥）                 │
└─────────────────────────────────────────────┘
```

- `llmops` 蓝图：约 460 条路由，覆盖应用/知识库/工作流/工具等全部业务
- `openapi` 蓝图：外部 API 调用路由，用于发布后的应用对外暴露

**设计意义**：同一套 Service 层同时服务两种调用方式，无需重复代码。

Agent 核心是一个 LangGraph 编译状态图，四个节点构成执行环路：

```
                    ┌─────────────────┐
                    │ preset_operation │  预处理（加载记忆、工具绑定）
                    └────────┬────────┘
                             │
              ┌──────────────▼──────────────┐
              │ long_term_memory_recall      │  长期记忆召回
              └──────────────┬──────────────┘
                             │
                    ┌────────▼────────┐
           ┌──────→│      llm        │  LLM 推理
           │       └────────┬────────┘
           │                │
           │      ┌─────────▼─────────┐
           │      │ tools_condition    │  是否需要调用工具？
           │      └──┬────────────┬───┘
           │    是   │            │  否
           │   ┌─────▼─────┐     │
           └───│   tools    │     ▼
               └───────────┘    END
```

### 4.2 ReACT vs FunctionCall 双模式

| 特性 | FunctionCallAgent | ReACTAgent |
|------|-------------------|------------|
| 工具调用方式 | LLM 原生 `bind_tools()` | Prompt 注入工具描述 + JSON 解析 |
| 适用模型 | 支持 TOOL_CALL 特性的模型 | 任何文本生成模型 |
| 工具检测 | LLM 返回 tool_calls 字段 | 解析输出前7字符是否为 `` ```json `` |
| 系统提示词 | 无额外模板 | REACT_AGENT_SYSTEM_PROMPT_TEMPLATE |

**亮点**：ReACTAgent 继承 FunctionCallAgent，仅重写 `_long_term_memory_recall_node` 和 `_llm_node` 两个方法，通过前7字符判断（`` ```json ``）区分工具调用与文本生成——让不支持原生工具调用的模型也能使用 Agent 能力。

### 4.3 事件队列流式输出

Agent 在子线程中执行，主线程通过 `AgentQueueManager` 的发布-订阅模式获取流式事件：

```python
# 子线程执行 Agent
thread = Thread(target=self._agent.invoke, args=(input,))
thread.start()

# 主线程监听事件队列
yield from self._agent_queue_manager.listen(task_id)
```

事件类型：`AGENT_START` → `LONG_TERM_MEMORY_RECALL` → `AGENT_THOUGHT` → `AGENT_ACTION` → `DATASET_RETRIEVAL` → `AGENT_MESSAGE` → `AGENT_END`

### 5.1 节点工厂映射

```python
NodeClasses = {
    NodeType.START:              StartNode,
    NodeType.END:                EndNode,
    NodeType.LLM:                LLMNode,
    NodeType.TEMPLATE_TRANSFORM: TemplateTransformNode,
```

### 5.2 四层图校验

工作流发布前执行严格校验，防止运行时错误：

| 校验层级 | 内容 | 算法 |
|---------|------|------|
| 节点校验 | ID/Title/Type 唯一性，Start/End 有且仅一个 | 集合去重 |
| 边校验 | ID 唯一，source/target 节点存在，无重复边 | 哈希表 |
| 图结构校验 | 从 Start 可达所有节点 | BFS 连通性检查 |
| 环检测 | 不允许存在环路 | Kahn 拓扑排序 |
| 数据引用校验 | inputs 引用的 node_id 在前置节点中，变量名存在 | 前驱遍历 |

### 5.3 特殊节点实现

- **CodeNode**：通过 AST 解析验证用户代码必须包含 `main(params)` 函数且仅有一个参数，然后 `exec()` 执行
- **LLMNode**：Jinja2 模板渲染变量注入 → 加载 LLM 实例 → 流式调用 `llm.stream()` → 聚合输出
- **DatasetRetrievalNode**：注入 `flask_app` 和 `account_id`，调用检索服务

**双输出格式**：

## 12. SSE 流式解析（前端）

**关键文件**：`src/utils/request.ts`

**亮点**：基于 ReadableStream 的递归流读取 + 缓冲区管理 + Promise.race 超时。

```typescript
// 核心流程
const reader = response.body?.getReader()
const decoder = new TextDecoder('utf-8')
let buffer = ''

const read = () => {
  reader?.read().then((result) => {
    if (result.done) { resolve(); return }

    buffer += decoder.decode(result.value, { stream: true })
    const lines = buffer.split('\n')

    // 解析 SSE 协议：event: xxx → data: xxx → 空行触发回调
    lines.forEach((line) => {
      if (line.startsWith('event:')) event = line.slice(6).trim()
      else if (line.startsWith('data:')) data = line.slice(5).trim()
      else if (line === '' && event && data) {
        onData({ event, data: JSON.parse(data) })  // 触发回调
        event = ''; data = ''
      }
    })

    buffer = lines.pop() || ''  // 保留未完成的最后一行
    read()  // 递归读取下一块
  })
}

// 超时保护
Promise.race([
  new Promise((_, reject) => setTimeout(() => reject('接口已超时'), 100000)),
  actualRequest
])
```

### Hooks 模式示例

```typescript
// 典型 Hook：封装分页加载 + loading + 数据累加
export const useGetAppsWithPage = () => {
  const loading = ref(false)
  const apps = ref<Array>([])
  const paginator = ref({ current_page: 1, page_size: 20, total_page: 0 })

  const loadApps = async (init = false, search = '') => {
    if (init) paginator.value.current_page = 1
    else if (paginator.value.current_page > paginator.value.total_page) return

    loading.value = true
    const resp = await getAppsWithPage({ ...paginator.value, search_word: search })

    if (init) apps.value = resp.data.list       // 首次：替换
    else apps.value.push(...resp.data.list)      // 翻页：追加

    paginator.value = resp.data.paginator
    paginator.value.current_page += 1
    loading.value = false
  }

  return { loading, apps, paginator, loadApps }
}
```

**复用效果**：Apps、Workflows、Datasets、Conversations、Tools 全部使用相同的分页 Hook 模式。

```typescript
const autoLayout = () => {

  // 将计算结果映射到节点 position
  nodes.forEach(n => {
    const pos = dagreGraph.node(n.id)
    n.position = { x: pos.x - pos.width/2, y: pos.y - pos.height/2 }
  })
}
```

## 核心设计模式速查

| 模式 | 应用位置 | 实现方式 |
|------|---------|---------|
| **依赖注入** | 后端全栈 | `injector` 库 + `@inject @dataclass` |
| **策略模式** | LLM提供商、检索策略 | `LanguageModelManager` + `providers.yaml` |
| **工厂模式** | 工作流节点创建 | `NodeClasses` 字典映射 |
| **状态机** | Agent/Workflow | LangGraph `StateGraph` → `CompiledStateGraph` |
| **发布-订阅** | Agent 流式输出 | `AgentQueueManager` + 线程间队列 |
| **模板方法** | 工作流节点基类 | `BaseNode.invoke()` 统一接口 |
| **Composition** | 前端业务逻辑 | `use-xxx.ts` Hooks 封装 |
| **分层架构** | 前后端均适用 | Handler→Service→Model / Views→Hooks→Services→Models |
